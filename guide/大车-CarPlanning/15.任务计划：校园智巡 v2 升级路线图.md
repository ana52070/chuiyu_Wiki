## 阶段 1：FAST-LIO2 独立编译与验证（3-4 天）

> **目标**：确认 FAST-LIO2 能在 Xavier NX 上跑起来，正确订阅 Livox Mid-360 和 IMU，输出合理的里程计数据。这个阶段不碰 EKF 和 Nav2。

### 任务 1.1：选择 ROS2 Humble 移植版本

- 搜索 GitHub 上 FAST-LIO2 的 ROS2 Humble 移植版，筛选标准：
    
    - 明确支持 ROS2 Humble
        
    - 有 Livox 点云格式支持（`livox_ros_driver2` 兼容）
        
    - 有近期维护（最近 6 个月内有 commit）
        
    - 有人在 ARM 平台（Jetson）上验证过
        
- 候选仓库（需要搜索确认最新状态）：
    
    - `Ericsii/FAST_LIO` (ROS2 分支)
        
    - `hku-mars/FAST_LIO` 官方是否有 ROS2 支持
        
    - 其他社区 fork
        
- **产出**：确定使用哪个仓库，fork 到自己的 GitHub
    

### 任务 1.2：在 Xavier NX 上编译

- 安装编译依赖：Eigen3, PCL, Sophus (如需)
    
- 处理可能的编译问题：
    
    - ARM 架构的特殊编译选项
        
    - PCL 版本兼容性
        
    - C++17 特性支持
        
    - livox_ros_driver2 消息类型引用
        
- 编译命令：
    

cd /ros2_ws/workspace  
git clone <选定的仓库> fast_lio2  
cd fast_lio2  
source /opt/ros/humble/setup.bash  
source /ros2_ws/workspace/livox/install/setup.bash  
colcon build --packages-select fast_lio

- **产出**：编译成功，`colcon build` 无 error
    

### 任务 1.3：配置 Livox Mid-360 参数

- 创建或修改 FAST-LIO2 的 YAML 配置文件：
    
    - `lidar_type` 设为 Livox
        
    - `scan_line` 设为 Mid-360 对应的值
        
    - 点云话题设为 `/livox/lidar`
        
    - IMU 话题设为 `/imu/data_raw`
        
- 配置 LiDAR-IMU 外参：
    
    - 初始值：平移 `[0.14, 0.0, 0.12]`（IMU 在 base_link 中心，LiDAR 在前方 0.14m、高 0.12m）
        
    - 旋转：取决于 IMU 和 LiDAR 的坐标系定义，需要确认
        
- IMU 参数：
    
    - 查阅底盘 IMU 的 datasheet（或根据经验设定 MEMS 典型值）
        
    - `acc_cov`, `gyr_cov`, `b_acc_cov`, `b_gyr_cov`
        
- **产出**：配置文件就绪
    

### 任务 1.4：室外静态测试

- 不启动 Nav2 和 EKF，只启动：底盘驱动 + Livox 驱动 + IMU + FAST-LIO2
    
- 启动命令（手动，不用 bringup.launch.py）：
    

# 终端 1：底盘驱动（提供 IMU）  
ros2 launch myslam_car driver.launch.py  
​  
# 终端 2：Livox 驱动  
ros2 launch carplanning_code mid360.launch.py  
​  
# 终端 3：FAST-LIO2  
ros2 launch fast_lio mapping.launch.py

- 检查项：
    
    - [ ]  FAST-LIO2 启动无 crash
    - [ ]  `ros2 topic hz /lio_odom` 显示 ~10Hz
    - [ ]  `ros2 topic echo /lio_odom --field pose.pose.position` 静止时位置基本不变（漂移 < 5cm/分钟）
    - [ ]  `ros2 topic echo /lio_odom --field pose.pose.orientation` 静止时 yaw 基本不变
    - [ ]  CPU 占用检查：`htop`，FAST-LIO2 进程 CPU 占用 < 50%

### 任务 1.5：室外动态测试

- 手动遥控机器人走一个矩形路径（约 5m × 5m），回到起点
    
- 记录 FAST-LIO2 的轨迹，检查：
    
    - [ ]  终点与起点的位置误差（理想 < 10cm）
    - [ ]  直线行驶时轨迹不弯曲
    - [ ]  转弯时 yaw 响应正确（左转 90° 实际显示 ~90°）
- 如果 IMU 频率不足（10Hz）导致里程计不准：
    
    - 选项 A：在 Xavier NX 上外接一个独立 IMU（如 BNO055 USB 或 ICM-42688 SPI）
        
    - 选项 B：尝试提高底盘 IMU 发布频率（修改底盘驱动固件/配置）
        
    - 选项 C：尝试 KISS-ICP（不需要 IMU）作为备选方案
        

### 阶段 1 完成标准

- [ ]  FAST-LIO2 在 Xavier NX 上编译通过并可运行
- [ ]  静止测试：1 分钟内位置漂移 < 5cm，yaw 漂移 < 1°
- [ ]  动态测试：5m×5m 矩形闭环误差 < 20cm
- [ ]  CPU 占用在可接受范围内（FAST-LIO2 < 50%）
- [ ]  输出话题格式确认（Odometry 消息的 frame_id、child_frame_id、协方差）

## 阶段 2：EKF 集成与改造（2-3 天）

> **目标**：把 FAST-LIO2 的输出接入 EKF，替代轮式里程计，与 RTK-GPS 联合融合。验证融合后的位姿在各种 GPS 条件下都正确。

### 任务 2.1：修改 ekf.yaml

- 核心改动：
    

# 旧配置  
odom0: /odom/relay  
odom0_config: [...vx, vy, vyaw...]  
imu0: /imu/relay  
imu0_config: [...vyaw...]  
​  
# 新配置  
odom0: /lio_odom  
odom0_config: [false,false,false,  # 不融合 LIO 位置（GPS 负责）  
               false,false,false,  
               true, true, false,  # vx, vy  
               false,false,true,   # vyaw  
               false,false,false]  
# 移除 imu0（IMU 已在 FAST-LIO2 内部融合）

- 调整 `sensor_timeout`：FAST-LIO2 输出 10Hz，`sensor_timeout` 可以收紧到 0.2s
    
- 调整 `frequency`：FAST-LIO2 10Hz → EKF 频率可以提到 10Hz
    
- 更新过程噪声协方差（FAST-LIO2 精度更高，可以适当降低过程噪声）
    
- 保留 `odom1: /odometry/gps` 的配置不变
    

### 任务 2.2：修改 bringup.launch.py

- 在 Livox 驱动之后（t=3s）添加 FAST-LIO2 的启动
    
- EKF 启动延迟从 t=5s 调整为 t=6s（给 FAST-LIO2 初始化时间）
    
- 决定是否保留 `covariance_relay.py`：
    
    - 如果完全移除轮式里程计输入 → 移除 relay
        
    - 如果保留轮式里程计作为备份 → 保留 relay 但可能改输入名
        
- 确保 FAST-LIO2 不发布与 EKF 冲突的 TF
    

### 任务 2.3：TF 冲突排查

- 检查 FAST-LIO2 是否默认发布 TF
    
- 如果发布，需要配置为不发布 TF（只发布 Odometry 消息）
    
- 确认 `odom → base_link` 的 TF 仍然由 EKF 或底盘驱动发布
    
- 运行 `ros2 run tf2_tools view_frames` 确认 TF 树正确
    

### 任务 2.4：融合效果验证

- 启动全部定位组件（不启动 Nav2）：底盘 + Livox + GPS + FAST-LIO2 + EKF + navsat
    
- 在 RViz2（远程机器）中检查：
    
    - [ ]  `map → odom` TF 正常发布
    - [ ]  `odom → base_link` TF 正常发布
    - [ ]  `/odometry/filtered` 输出频率 ~10Hz
    - [ ]  静止时 RViz 中机器人位置不跳动
    - [ ]  手动遥控移动时，RViz 中的运动方向与真实世界一致
    - [ ]  GPS 质量变化时（如果能制造这种条件），定位不会剧烈跳变

### 阶段 2 完成标准

- [ ]  EKF 稳定融合 FAST-LIO2 + GPS，输出频率 ≥ 10Hz
- [ ]  TF 树无冲突，`ros2 run tf2_tools view_frames` 输出正确
- [ ]  手动遥控时 RViz 中机器人运动与真实世界方向一致
- [ ]  EKF 不再报 "Failed to meet update rate" 警告
- [ ]  静止时 RViz 位置稳定（无明显漂移或跳动）

## 阶段 3：导航联调与标定（3-4 天）

> **目标**：接入 Nav2 全栈，进行完整的导航测试。调整参数使路径跟踪准确、避障有效。

### 任务 3.1：Nav2 频率调整

- 由于 EKF 现在跑 10Hz（比 v1 的 5Hz 更快），可以考虑适当提高 Nav2 频率：
    

controller_frequency: 10.0     # 保持 10Hz 或提到 15Hz  
expected_planner_frequency: 5.0  
bt_loop_duration: 100          # 保持  
transform_tolerance: 0.3       # 从 0.5s 收紧到 0.3s（EKF 更快了）

- 但要监控 CPU，确保不过载
    

### 任务 3.2：yaw_offset 标定验证

- 这是一个遗留问题：`yaw_offset: 1.5708` (π/2) 是否正确？
    
- 标定方法：
    
    1. 将机器人放在已知方向（如面朝正北，用指南针确认）
        
    2. 启动系统，等 GPS 稳定
        
    3. 在 RViz 中查看机器人在 map 坐标系中的朝向
        
    4. 如果 map 坐标系中机器人朝北 → yaw_offset 正确
        
    5. 如果偏了 N 度 → 调整 yaw_offset
        
- 注意：有了 FAST-LIO2 之后，yaw_offset 对跟踪精度的影响变小了（LIO 提供高精度 yaw），但它仍然影响全局坐标系的对齐
    

### 任务 3.3：短距离导航测试（5m）

- 在空旷区域发送 5m 距离的导航目标
    
- 观察：
    
    - [ ]  全局路径规划正确
    - [ ]  机器人沿路径行驶，不振荡
    - [ ]  到达目标点附近后停止（误差 < 0.5m）
    - [ ]  代价地图不旋转
    - [ ]  DWB 不报 "No valid trajectories"

### 任务 3.4：中距离导航测试（20m）

- 在有障碍物的环境中发送 20m 距离的导航目标
    
- 观察：
    
    - [ ]  遇到障碍物时重新规划路径
    - [ ]  绕过障碍物后回到正确路径
    - [ ]  不会原地打转
    - [ ]  不会进入 recovery 循环（spin/backup/wait）

### 任务 3.5：DWB 控制器参数调优

如果跟踪精度仍不满意，调整 DWB 参数：

- `PathAlign.scale` / `GoalAlign.scale`：路径对齐权重
    
- `PathDist.scale` / `GoalDist.scale`：路径/目标距离权重
    
- `max_vel_x`, `max_vel_theta`：速度限制
    
- `vx_samples`, `vtheta_samples`：采样密度（CPU 允许时可以提高）
    

### 阶段 3 完成标准

- [ ]  5m 导航测试：成功到达，路径跟踪误差 < 30cm
- [ ]  20m 导航测试：成功到达，能避开至少一个障碍物
- [ ]  代价地图方向与真实世界一致（不旋转）
- [ ]  不出现 recovery 循环

## 阶段 4：鲁棒性测试（2-3 天）

> **目标**：验证系统在 GPS 降级、环境变化等非理想条件下的表现。

### 任务 4.1：模拟 GPS 降级

- 方法 1：在导航过程中手动断开 NTRIP（让 GPS 从 RTK 降级到 SPP）
    
- 方法 2：走到建筑物遮挡区域
    
- 方法 3：在 EKF 配置中临时增大 GPS 协方差模拟精度下降
    
- 观察：
    
    - [ ]  GPS 降级瞬间，机器人不会剧烈跳动
    - [ ]  GPS 降级后，导航继续工作（FAST-LIO2 主导）
    - [ ]  短期（1-2 分钟）内路径跟踪仍然可接受
    - [ ]  GPS 恢复后，位置自动修正，无剧烈跳变

### 任务 4.2：GPS 完全丢失测试

- 在导航过程中完全关闭 GPS 驱动节点
    
- 观察纯 FAST-LIO2 里程计下的导航持续时间和精度
    
- 记录：从 GPS 丢失到导航显著偏差的时间
    

### 任务 4.3：复杂环境导航

- 在有多个障碍物（柱子、停放的车辆、行人）的环境中导航
    
- 在道路拐角处导航
    
- 在窄通道（宽度 ~1.5m）中导航
    
- 记录成功率和失败模式
    

### 任务 4.4：长时间运行测试

- 连续运行系统 30 分钟以上
    
- 监控：
    
    - [ ]  内存使用是否持续增长（FAST-LIO2 的 ikd-Tree 内存泄漏？）
    - [ ]  CPU 占用是否稳定
    - [ ]  EKF 是否持续正常输出
    - [ ]  导航精度是否随时间下降

### 阶段 4 完成标准

- [ ]  GPS 降级测试：导航不崩溃，可继续工作 ≥ 2 分钟
- [ ]  GPS 完全丢失：系统不 crash，里程计至少维持 1 分钟
- [ ]  复杂环境：成功率 ≥ 70%（允许需要多次尝试）
- [ ]  长时间运行：30 分钟无 crash，内存/CPU 稳定

## 风险与备选方案

|风险|概率|影响|备选方案|
|---|---|---|---|
|FAST-LIO2 ROS2 Humble 编译失败|中|高|尝试其他 fork；或用 Docker 封装 ROS1 版本 + ros1_bridge|
|底盘 10Hz IMU 不够 FAST-LIO2 使用|中|高|换 KISS-ICP（不需要 IMU）；或增加独立 IMU|
|Xavier NX CPU 不够跑全栈|低|高|增加一块计算平台（如你提到的方案）；或用轻量级替代如 KISS-ICP|
|LiDAR-IMU 外参标定困难|中|中|使用 LI-Init 自动标定工具；或手动测量 + 迭代调整|
|FAST-LIO2 在特定环境下精度差|低|中|增大 EKF 中 LIO 的协方差；或在开阔区域加大 GPS 权重|
