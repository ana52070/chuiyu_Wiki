# 项目概述：RTK + LiDAR 里程计无地图自主导航系统（v2）

> **文档版本**: 2026-06-21 v2  
> **分支**: `hardware`  
> **与 v1 的核心差异**:
> 
> 1. 引入 FAST-LIO2 LiDAR 惯性里程计，替代轮式里程计作为 EKF 主要运动估计来源
> 2. 计算平台从 NVIDIA Jetson Xavier NX 迁移至 Intel N100 Mini PC

---

## 1. 项目目标

基于 RTK-GPS + LiDAR 里程计的**无预建地图**园区自主导航。机器人不需要事先 SLAM 建图，通过 FAST-LIO2 提供高精度相对定位 + RTK-GPS 提供绝对位置修正 + 3D LiDAR 实时感知 + Nav2 导航栈，实现点到点自动驾驶。

核心特性：

- **无地图导航（Mapless Navigation）**：使用 STVL 将 3D LiDAR 点云直接投影为 2D 代价地图，无需预建静态地图
- **双层定位融合**：FAST-LIO2 提供 10Hz 高精度局部里程计（scan matching + IMU 紧耦合），RTK-GPS 提供绝对位置修正，EKF 融合两者
- **弱 GPS 鲁棒性**：GPS 降级或短暂丢失时，FAST-LIO2 独立维持定位精度（分钟级无明显漂移），GPS 恢复后自动修正累积漂移
- **低速园区场景**：最大速度 0.26 m/s，适用于校园/园区环境

---

## 2. v1 → v2 架构演进：为什么要改？

### 2.1 v1 的结构性缺陷

v1 系统的传感器融合架构：

```
轮式里程计 (10Hz, 协方差注入) ──→ EKF ←── RTK-GPS (5-20Hz)
低成本 IMU (10Hz, 协方差注入) ──→     ←── navsat_transform
```

这个架构存在一个根本性问题：**在两次 GPS 更新之间（50-200ms），EKF 只能依赖轮式里程计和低成本 IMU 来估计机器人运动**。

Yahboomcar 的轮式里程计问题：

- 没有编码器校准（轮径、轮距未标定）
- 四轮差速底盘存在滑动，yaw 积分漂移严重
- 协方差矩阵原始值全零，需要外部 relay 注入估计值
- 速度估计不确定度约 ±5cm/s（对于最大速度 26cm/s 的车来说，这是 ~20% 的误差）

底盘内置 IMU 问题：

- 低成本 MEMS（类似 MPU6050），陀螺仪零偏 ±1-5°/s
- 协方差矩阵原始值全零
- 受底盘振动影响严重

后果：DWB 控制器每 100ms 决策一次速度指令，它看到的"当前位姿和速度"来自 EKF，而 EKF 在 GPS 空窗期只有这两个低质量传感器。GPS 来了位置被拉到正确位置，GPS 走了就开始歪，下次 GPS 来了又拉回来——这就是振荡和跟踪不准的根源。

### 2.2 v1 的工程痛点：Xavier NX 平台限制

除了传感器架构问题，Xavier NX 作为计算平台本身也带来了大量工程开销：

- JetPack 基于 Ubuntu 20.04，无法原生安装 ROS2 Humble，必须通过 Docker 部署
- ARM 架构导致大量依赖包缺失或需要源码编译（如 STVL、robot_localization 的 ARM 构建）
- NVMe SSD 故障需要 reflash JetPack + rootOnNVMe 迁移，流程繁琐
- dustynv 社区 Docker 镜像的各种限制和不稳定性
- CPU 资源紧张（6 核 ARM），20+ 节点全部竞争，频繁触发 "Failed to meet update rate"
- 384 个 CUDA 核心在本项目中完全闲置（FAST-LIO2、Nav2、EKF 均为纯 CPU 运算）

### 2.3 v2 的双重改进

**改进一：引入 FAST-LIO2 LiDAR 里程计**

利用已有的 Livox Mid-360 进行点云帧间匹配，输出高精度的 6DOF 里程计：

```
Livox Mid-360 ──→ FAST-LIO2 ──→ LiDAR 里程计 (10Hz, 高精度) ──→ EKF ←── RTK-GPS
底盘 IMU ─────→ (FAST-LIO2 内部紧耦合)                              ←── navsat_transform

Livox Mid-360 ──→ STVL 代价地图 (与 v1 相同)
```

**改进二：计算平台迁移至 N100 Mini PC**

用 Intel N100 Mini PC（Ubuntu 22.04）替换 Xavier NX：

- x86 架构，ROS2 Humble 原生安装，所有依赖 `apt install` 一行搞定
- FAST-LIO2 在 x86 上编译无 ARM 兼容性问题
- N100 4 核 x86 的单核性能与 Xavier NX 6 核 ARM 相当，但编译和依赖管理大幅简化
- TDP 仅 6W（Xavier NX: 10-15W），对车载供电更友好
- 机器上已有一台（之前跑 OpenClaw），直接复用

## 3. 硬件平台

### 3.1 计算平台

|项目|v1 (旧)|v2 (新)|
|---|---|---|
|型号|NVIDIA Jetson Xavier NX|Intel N100 Mini PC|
|CPU|6 核 ARM Carmel|4 核 x86 Alder Lake-N|
|内存|8GB|8/16GB|
|操作系统|Ubuntu 20.04 (JetPack)|Ubuntu 22.04|
|ROS2|Humble (Docker 部署)|Humble (原生安装)|
|TDP|10-15W|6W|
|GPU|384 CUDA 核心 (未使用)|集显 (未使用)|

### 3.2 传感器与执行器

|组件|型号|接口|在 v2 中的角色|
|---|---|---|---|
|LiDAR|Livox Mid-360|以太网 (192.168.1.x)|**双重角色**：FAST-LIO2 里程计 + STVL 代价地图|
|RTK-GPS|UM982 双天线|USB 串口 (`/dev/rtk1`)|绝对位置修正，防止长期漂移|
|IMU|Yahboomcar 底盘内置|USB 串口 (通过 STM32)|喂给 FAST-LIO2 内部融合|
|底盘|Yahboomcar（四轮差速）|USB 串口 (`/dev/stm32`)|执行控制指令，轮式里程计降为备份|

### 3.3 接线迁移

从 Xavier NX 到 N100 的接线完全相同，只是 USB 端口物理位置变了：

```
N100 Mini PC
├── USB-A ─── UM982 GPS (/dev/rtk1, udev 绑定)
├── USB-A ─── STM32 底盘 (/dev/stm32, udev 绑定)
└── 以太网 ─── Livox Mid-360 (静态 IP: 192.168.1.x)
```

udev 规则需要迁移——将 Xavier NX 上的 `/etc/udev/rules.d/` 规则复制到 N100，但 serial ID 需要根据 N100 的实际 USB 端口更新。

Livox 需要 N100 的以太网口配置静态 IP（同网段 `192.168.1.x`），与 Xavier NX 上的配置方式相同。

### 3.4 传感器安装位置

与 v1 相同（硬件不变，只换主控）：

```
        前 (+X)
         ↑
         |   LiDAR (0.14, 0, 0.12)
         |
    ─────┼─────→ 右 (+Y)
         |
    GPS2 ●──┼──● GPS1     ← GPS 天线在后方 (-0.14)
         |
       后
```

GPS 基线：16cm，左右布局，`yaw_offset = π/2 (1.5708 rad)`。

### 3.5 供电设计

N100 Mini PC 典型供电为 12V DC。车载 4S LiPo（14.8-16.8V）需要通过 DC-DC 降压模块降至 12V。N100 的 6W TDP + 外设总功耗约 15-20W，比 Xavier NX 时期的 25-30W 更低，现有电池续航会更长。

### 4.2 Livox Mid-360 的双重角色

v2 中 Livox Mid-360 的点云 `/livox/lidar` 同时被两个系统订阅：

1. **FAST-LIO2**：用于帧间点云匹配，计算机器人的精确相对运动（里程计）。FAST-LIO2 内部维护一个增量式的 ikd-Tree 局部地图，用于高效的最近邻搜索。这个局部地图不是 Nav2 意义上的"导航地图"，它只用于 scan matching 计算里程计。
    
2. **STVL（SpatioTemporalVoxelLayer）**：用于构建代价地图，识别障碍物。这与 v1 完全相同。STVL 订阅同一个点云话题，将 3D 点云投影为带时间衰减的体素，再压缩为 2D costmap 给 Nav2 使用。
    

两个系统独立运行，互不干扰。ROS2 的话题发布-订阅模型天然支持一个话题被多个节点同时订阅。

### 4.3 EKF 传感器融合配置（v2）

v2 的 `ekf.yaml` 核心变化：

```yaml
# ====== v2 EKF 配置 ======
frequency: 10.0              # N100 算力充足，EKF 可以跑 10Hz
two_d_mode: true
sensor_timeout: 0.5

# --- odom0: FAST-LIO2 LiDAR 里程计（替代轮式里程计）---
odom0: /lio_odom
odom0_config: [false, false, false,   # 不融合 LIO 的绝对位置（交给 GPS）
               false, false, false,
               true,  true,  false,   # vx, vy
               false, false, true,    # vyaw
               false, false, false]

# --- odom1: GPS 绝对位置 + 航向（与 v1 相同）---
odom1: /odometry/gps
odom1_config: [true,  true,  false,   # x, y (绝对位置)
               false, false, true,    # yaw (双天线航向)
               false, false, false,
               false, false, false,
               false, false, false]

# 不再需要单独的 imu0 输入
# IMU 数据已在 FAST-LIO2 内部完成紧耦合融合
# FAST-LIO2 的输出里程计已经包含了 IMU 的高频信息
```

### 4.4 TF 坐标变换树（v2）

```
map ──(EKF)──► odom ──(FAST-LIO2 或 EKF)──► base_link
                                                │
                                ┌───────────────┼───────────────┐
                                │               │               │
                          livox_frame      base_footprint      gps
                         (静态 TF)          (底盘驱动)      (静态 TF)
```

FAST-LIO2 需要配置为只发布 Odometry 消息（`/lio_odom`），不发布 TF，避免与 EKF 的 TF 冲突。EKF 统一负责发布 `map → odom` 变换。

### 4.5 节点启动时序（v2）

```
t=0s   ─── 底盘驱动 (串口, 提供 /cmd_vel 接收)
t=0s   ─── GPS/LiDAR 静态 TF
t=2s   ─── Livox Mid-360 驱动 (需要初始化)
t=2s   ─── UM982 RTK 驱动
t=3s   ─── FAST-LIO2 (等 LiDAR 点云开始发布)
t=5s   ─── EKF (等 FAST-LIO2 输出稳定)
t=8s   ─── navsat_transform (等 EKF 先跑起来)
t=10s  ─── Nav2 导航栈
```

### 4.6 节点列表与职责

|节点|输入|输出|职责|
|---|---|---|---|
|`driver_node` (底盘)|`/cmd_vel`|`/odom`, `/imu/data_raw`, TF `odom→base_footprint`|电机控制、里程计、IMU|
|`livox_ros_driver2`|以太网|`/livox/lidar` (PointCloud2)|LiDAR 数据采集|
|`um982_ros2_driver`|USB 串口|`/gps/fix`, `/gps/utmpos`|RTK-GPS 数据|
|**`fast_lio2`** (新增)|`/livox/lidar`, `/imu/data_raw`|`/lio_odom` (Odometry)|**LiDAR 惯性里程计**|
|`ekf_node`|`/lio_odom`, `/odometry/gps`|TF `map→odom`, `/odometry/filtered`|传感器融合|
|`navsat_transform`|`/gps/fix`, `/odometry/filtered`|`/odometry/gps`|GPS 经纬度→UTM 坐标|
|Nav2 全栈|TF, `/livox/lidar`|`/cmd_vel`|路径规划、轨迹跟踪、避障|

## 5. 传感器精度对比

|传感器|位置精度|yaw 精度|更新频率|长期漂移|在 v2 中的角色|
|---|---|---|---|---|---|
|轮式里程计|±5cm/s 速度|±5°/s gyro drift|10Hz|严重|**降为备份/移除**|
|低成本 IMU|N/A|±1-5°/s zero-bias|10Hz|严重|**喂给 FAST-LIO2**|
|FAST-LIO2|~1cm 相对|~0.1° 相对|10Hz|轻微（分钟级）|**主要运动估计**|
|RTK-GPS (Fixed)|~1.4cm 绝对|~1° (双天线16cm)|5-20Hz|无|**绝对位置锚定**|

## 6. GPS 降级处理

|GPS 状态|Cov_H 范围|系统行为|导航效果|
|---|---|---|---|
|RTK Fixed|< 0.001 m²|GPS 高权重修正，系统最优|最佳|
|RTK Float|0.001 - 0.25 m²|GPS 中等权重，LIO 为主|良好|
|DGPS / SPP|0.25 - 10 m²|GPS 低权重，LIO 主导|可接受（短期）|
|无 GPS|无数据|纯 LIO 运行|可接受（数分钟）|

EKF 根据 `/odometry/gps` 中的协方差矩阵自动调整 GPS 融合权重。UM982 驱动已经在 `position_covariance` 中填入真实的 `bestpos_latstd²` 和 `bestpos_lonstd²`。

## 10. 注意事项

1. **N100 没有 NVIDIA GPU**：本项目不需要 GPU，所有运算纯 CPU。如果未来需要运行视觉推理模型（如 YOLO），需要用 OpenVINO 而非 CUDA
2. **FAST-LIO2 需要 IMU 数据**：它内部做 IMU 预积分，必须订阅 `/imu/data_raw`。底盘的 10Hz IMU 可能不够，需实测验证
3. **LiDAR-IMU 外参需要准确**：安装误差会直接影响里程计精度，需要标定
4. **N100 的以太网口需要配置静态 IP**：与 Livox Mid-360 通信需要同网段（192.168.1.x）
